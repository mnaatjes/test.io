## Question 1
**Question:** Where can you submit a Bug Reproduction? (select 3 answers)
1. In the "Similar bug reports" list on the right-hand side of the bug report submission form
2. On the Test Overview page
3. On your Tester Dashboard under "Bugs you can reproduce" section
4. When viewing another tester's bug report

**Answer:** 1, 3, and 4.
**Reasoning:** The "Reproduction Feature Notes" and "Reproduction Feature Quick Reference" explicitly list these four ways to find reproductions: Dashboard (3), Inside Bug Reports (4), Similar Bug Reports List (1), and the Bugs List toggle. The Test Overview page (2) is not mentioned as a direct place to submit reproductions.

## Question 2
**Question:** What is a Negative Reproduction?
1. A Reproduction that shows that you cannot reproduce the described bug on your device after following the same steps
2. A Reproduction that shows that you can reproduce the described bug on your device after following the same steps
3. There are no Negative Reproductions on the test IO platform
4. A poorly documented Reproduction

**Answer:** 1. A Reproduction that shows that you cannot reproduce the described bug on your device after following the same steps
**Reasoning:** As defined in the "Bug Reproductions Notes", a negative reproduction is when you follow the same steps as the original tester but cannot reproduce the bug, showing instead that the function works on your device.

## Question 3
**Question:** What are the requirements for the Bug Reproduction screencast on the Test IO platform? (select 3 answers)
1. The screencast should be at least 1 minute long.
2. The screencast should be no longer than 15 seconds in most cases
3. Showing the action that triggers the bug is enough in most cases
4. The screencast must show the current date

**Answer:** 2, 3, and 4.
**Reasoning:** The "Bug Reproductions Notes" state that a screencast is required, should usually be no longer than 15 seconds, and must include the current date. Showing only the triggering action is typically sufficient.

## Question 5
**Question:** In which cases can you submit a Bug Reproduction? (select 2 answers)
1. The bug report has not been reviewed by the Team Leader yet and there is an available Reproduction spot.
2. The test is still running.
3. It is your own bug.
4. The bug report has already been reviewed by the Customer.

**Answer:** 1 and 2.
**Reasoning:** Visibility criteria in the "Bug Reproductions Notes" specify that reproductions are available while the test is running (2) and only for reports not yet reviewed by the customer (which excludes 4). They are for other testers' bugs (excludes 3). The "Similar bug reports" rule confirms you can reproduce a bug before TL review if it appears while you are filing.

## Question 6

1. The date is not included whereas every Reproduction attachment should show the current date.
2. The URL is not shown whereas every Reproduction attachment should show the URL.
3. It is too short.

## Question 7
**Question:** Would this Reproduction be accepted by test IO?
An app crash reproduced on an Apple iPhone 11 Pro Max in Safari.
1. The reproduction would be accepted by test IO.
2. The Reproduction would be rejected as the wrong device is selected and a crash log file is missing.
NOTE: Example Image shows Multiple devices with Macbook Air and google chrome browser selected. "Upload a screenshot or screencast" has file "reproduction.mp4"

**Answer:** 2. The Reproduction would be rejected as the wrong device is selected and a crash log file is missing.
**Reasoning:** Based on the "Bug Reproductions Notes", when reproducing an app crash, uploading a crash log file is mandatory in addition to the screencast. Furthermore, the selected environment must match the device used. In this case, selecting MacBook/Chrome for an iPhone/Safari bug is incorrect and would lead to rejection.
