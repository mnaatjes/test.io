# Bug Reporting Reference Notes

High-level overview of Test IO bug reporting standards for documenting functional issues precisely and professionally. Adhering to these rules ensures reports are understood and accepted by Team Leaders.

## Standard Title Formula
*   **Formula:** What? + Where? + When?
*   Summarize the problem clearly without needing to read the full report.
*   Describe what **is** happening (e.g., "'Checkout' button navigates to Error 500") rather than what **is not** happening.
*   Include specific conditions, user types, or inputs.

## Steps to Reproduce
*   **First Step:** Access the landing page URL or open the specific mobile app.
*   **Middle Steps:** Separate distinct actions like buttons pressed or data entered.
*   **Last Step:** The specific action that triggers the bug (never use "Observe").
*   Keep steps as general as possible and use the minimum number required.

## Results and Environment
*   **Actual Result:** Detailed explanation of what happened after the last step. Avoid simple opposites of the expected result.
*   **Expected Result:** Describe correct behavior if the bug did not exist, detailing pages loaded or data saved.
*   **Environment:** Select the exact device/browser from your invitation list. One environment per report.

## Professionalism and Rules
*   No placeholders like "will edit later."
*   Use realistic inputs; avoid "asdf" or "test123."
*   Delete mistakes only **before** Team Leader review.
*   Feature selection and severity must be set before filling the report.
